Git 工作流与团队协作规范：从分支策略到 Code Review

发表于 2025-06-23 分类于 Git 阅读次数：
系统梳理 Git 团队协作的最佳实践：分支策略（Git Flow / GitHub Flow）、Commit 规范（Conventional Commits）、Code Review 清单、Git Hooks 自动化

前言

很多团队用 Git 只停留在 add → commit → push 的阶段。一旦多人协作，就会出现：commit 信息乱七八糟、分支混乱、合并冲突不断、上线回滚找不到版本。

本文从分支策略、提交规范、Code Review、自动化四个维度，建立一套可落地的 Git 协作规范。

一、分支策略

1.1 GitHub Flow（推荐）

最简单、最适合持续交付的分支模型：

规则：

  • main 始终是可部署的状态
  • 新功能从 main 拉出功能分支（feat/xxx）
  • 通过 Pull Request 代码审查后合并
  • 合并后立即部署

适合： 小团队、SaaS 产品、持续部署的项目。

1.2 Git Flow（适合发版周期长的项目）

分支类型：

分支 命名 来源 合并到
主分支 main - -
开发分支 develop main main
功能分支 feat/xxx develop develop
发布分支 release/x.x develop develop + main
修复分支 hotfix/x.x.x main develop + main

适合： 有固定发版周期、需要维护多个历史版本的项目。

二、Commit 规范

2.1 Conventional Commits（约定式提交）

类型：

类型 说明 语义化版本影响
feat 新功能 次版本号 +1
fix Bug 修复 修订号 +1
BREAKING CHANGE 破坏性变更 主版本号 +1
refactor 重构（非修复非功能） 无
docs 文档 无
style 格式（空格、分号等） 无
chore 构建/工具变更 无
test 测试 无
perf 性能优化 无

3.2 Code Review 清单

审查者在 CR 时应检查：

类别 检查项
✅ 逻辑正确性 代码是否能正常工作？边界条件是否处理？
✅ 安全性 用户输入是否校验？SQL 注入/XSS 防护？
✅ 性能 有没有不必要的重复计算？数据库查询是否优化？
✅ 可维护性 命名是否清晰？代码是否过于复杂？
✅ 测试覆盖 新功能是否包含测试？
✅ 错误处理 错误是否被正确处理而不是静默吞掉？
❌ 不要纠结 代码风格（有格式化工具）

3.3 避免过大的 PR

原则： 每个 PR 只做一件事。一个 PR 既加功能又重构还修 Bug = 噩梦。

总结：团队 Git 规范速查表

环节 规范 工具/配置
分支命名 feat/xxx / fix/xxx / chore/xxx 团队约定
Commit 信息 feat(scope): 描述 commitlint
代码检查 commit 前自动 lint husky + lint-staged
代码审查 PR < 300 行，附模板 GitHub / GitLab
合并策略 squash merge（保持主分支历史干净） 仓库设置
版本发布 语义化版本 + CHANGELOG standard-version
